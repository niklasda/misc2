Vi har en Model-View-Presenter-lösning.

Databasen
Databasen är Caché (en post-relationsdatabas), eftersom ingen antas ha den installerad kan man snabbt, med hjälp av spring.net byta ut den mot en mock-databas (med hårdkodad data).

Presenter
Presenter projektet innehåller presenters så klart. presenterar data från Model i View.
Presentern använder spring.net för att välja databastyp. 

View
View, webprojektet innehåller konfigurationen för spring.net och databaskoppling
aspx sidorna anropar bara presenters

Model
Model innehåller mappen Generated, som innehåller kod genererad av Cachés kodgenerator
Entities innehåller data för databindningen, de får sin data från objekter i Generated
Model innehåller också interfacet som implementeras av databashanterarna

DatabaseHandler
Handlers är de tillgängliga databashanterarna man kan välja mellan. Skulle nprmalt bara innehålla en.
Database innehåller skript för Caché databasen. Databasen innehåller Parent och Child object där Child object hör till Parent objekt.
Dvs om man hämtar ut ett Parent objekt fattar databasen att den ska hämta Child objekten också och stoppa dem i en lista.

Nya
Den nya lösningen implementerar MVP
Det finns ingen intressant kod i View
All kod ligger i vanliga assemblies och är lätt att unit-testa
IoC används (spring.net), tex lätt att mocka bort database eller ersätta den med en annan
använder interface
använder objektdatabas med samma klasser som domänmodellen
har en windows klient för att administrera databasen

Gamla
Den gamla lösningen var monolitisk, svårtestad, inga interface
model och controller sammanblandad
dålig modularisering och återanvändbarhet
inga kända patterns
använder tråkig relationsdatabas
